# ThreadSanitizer suppressions for the Orkige TSan gate (Docs/sanitizers.md).
#
# POLICY: this file suppresses ONLY races the engine cannot fix - third-party
# worker threads whose internal synchronisation TSan cannot see through. Every
# entry names its non-Orkige source and why. A race in Orkige's OWN code is a
# real bug: it is fixed, never added here. Wired in via
# TSAN_OPTIONS=suppressions=Util/tsan_suppressions.txt.
#
# The render-backend entries bite only the instrumented INTEGRATION/windowed
# runs (which spin up the render backend threads). The Jolt entries also bite
# the headless UNIT gate: the physics tests drive Jolt, which runs its OWN
# worker-thread job system - third-party threads the unit gate does touch.

# Ogre-Next spreads scene-graph update and frustum culling across its own
# worker-thread pool (SceneManager::_updateWorkerThread / cullFrustumExecute)
# with an internal Barrier the engine does not own - the reported cross-thread
# accesses are inside Ogre and cannot be resolved from Orkige code.
race:Ogre::SceneManager::updateWorkerThread
race:Ogre::SceneManager::cullFrustumExecute
race:Ogre::SceneManager::updateAllTransformsThread
race:Ogre::SceneManager::updateAllBoundsThread

# The clustered-forward light-collection pass (ForwardClustered) fills its grid
# from Ogre worker threads over the same internal Barrier - third-party.
race:Ogre::ForwardClustered

# Ogre-Next's low-level threading primitives (the pool/barrier that drive the
# above); suppress at the primitive so any future Ogre threaded stage is covered
# without a fresh entry - all strictly inside the third-party render backend.
race:Ogre::ThreadHandle
race:Ogre::Barrier

# Jolt (JPH::) runs physics work across its OWN job-system worker threads over
# lock-free atomics TSan cannot see through; the engine only drives Jolt from
# the main thread (engine_physic/PhysicsWorld) and drains contact events through
# an owned mutex queue (that queue is Orkige code, in the Orkige:: namespace, and
# is NOT suppressed - a race there would be a real bug). Any race sited ENTIRELY
# within the JPH:: namespace is Jolt-internal by construction (Orkige runs no
# code on Jolt's worker threads), so suppress at the namespace - the simulation
# step reaches many Jolt-internal sites (job system, scratch allocator, lock-free
# islands/collision containers) and chasing each one is both fragile and
# needless. Observed sites include JPH::JobSystem::Job::Execute and
# JPH::TempAllocatorImpl::Allocate. Jolt is STATICALLY linked, so a
# `called_from_lib` entry would not match - the symbol-namespace match is the
# working form.
race:JPH::

# OpenAL Soft runs its audio mixer on its OWN thread (even the null backend used
# in tests), racing internally on its lock-free voice-change list
# (alu.cpp ProcessVoiceChanges and siblings). Unlike physics, the sound tests
# carry NO Orkige-owned threading to protect - the engine drives OpenAL entirely
# from the main thread (engine_sound/SoundManager; music refill is main-thread in
# SoundManager::update), so the one test that spins a real AL device
# (SoundManagerDestructorTearsDownAL) is EXCLUDED from the TSan unit gate at the
# preset level (CMakePresets.json unit-*-tsan filter) rather than chasing OpenAL's
# internal race sites here. Its AL-teardown memory safety is still covered by the
# ASan gate. No race:AL... entry is needed as a result.
